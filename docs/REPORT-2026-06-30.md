## Key Findings

### 1. aider multifile failures resolved
Previous runs (2026-06-18) showed aider failing js-03 (multifile cache), js-04 (multifile rename), and js-06 (self-verify lint) consistently. The 2026-06-29 overnight run shows **100% across all 11 cases on all capable models** — no `--file` hints needed. This appears to be a model improvement (better instruction following) rather than an adapter change.

### 2. MoE dominates dense at all scales
Every MoE model in the suite outperforms dense models of comparable total parameter count:
- qwen3.6-35b-a3b (3.6B active) > qwen3.6-27b (27B dense) on speed (3–5× faster)
- gemma-4-26b-a4b-qat (4B active) > glm-4.7-flash (30B dense GGUF) on both speed and accuracy
- Rule: **always prefer MoE when available**

### 3. devstral opencode/openhands collapse is model-specific
devstral-small-2-2512 collapses to 37% on opencode and openhands (same as LMS-incompatible caveman). aider/goose/hermes/pi work fine on devstral (86–94%). The pattern suggests devstral doesn't follow the tool-call schema expected by opencode/openhands. Removed from active rotation.

### 4. glm-4.7-flash GGUF is too slow for benchmarking
Dense 30B GGUF on Apple Silicon generates ~1–3 tok/s. Most adapters hit the 300s timeout on every non-trivial case. codex (65%) is the only viable adapter because it batches the task in fewer round-trips. Not a configuration error — the model needs either MLX backend or higher timeout.

### 5. qwen3.6-27b re-added as LMS GGUF performs well
Previously removed (2026-06-12) in MLX form at 35.7% due to thinking token leakage and extreme slowness. Re-added as LMS GGUF on 2026-06-28: 93% across 6 tested adapters. codex/goose/interpreter all 100%. Only partial adapter sweep completed.

## New This Run

### nanocoder adapter (v1.28.1)

Full sweep across 7 models (6/7 complete as of report time):

| Model | Score | Notes |
|-------|-------|-------|
| qwen/qwen3.6-35b-a3b | 35/36 (97%) | Only js-04 fails (import not updated in consumer) |
| google/gemma-4-26b-a4b-qat | 36/38 (95%) | js-01 partial; all others PASS |
| qwen/qwen3.6-27b | 34/38 (89%) | bash-01 0/4 (execute_bash blocked); JS 100% |
| mistralai/devstral-small-2-2512 | 33/38 (87%) | bash-01 1/4, smoke-01 0/2; JS 100% |
| qwen/qwen3.5-9b | 24/34 (71%) | XML tool-call fallback — smoke-01/02, ts-01, js-05 all error(1) |
| zai-org/glm-4.7-flash | 12/34 (35%) | All 300s timeout — GGUF too slow for multi-turn tool calls |
| qwen/qwen3-coder-30b | 36/38 (95%) | bash-01 4/4 PASS — only model to pass without execute_bash; smoke-01 only failure |

**Key finding — execute_bash barrier:** nanocoder requires interactive approval for bash tool calls. In headless bench runs, `bash-01-topwords` (the only case that needs `chmod +x`) fails for most models. All JS cases (js-01–06) and most smoke cases pass cleanly. This is by design per the adapter-approval-policy (no auto-approve flags).

**Key finding — XML tool-call fallback (qwen3.5-9b):** qwen3.5-9b emits XML-format tool calls that nanocoder silently ignores. Affects smoke-01/02, ts-01, js-05; bash and non-edit JS cases still pass. Other models (35b-a3b, gemma, devstral, 27b) use proper structured JSON tool calls.

- **Integration:** `NANOCODER_PROVIDERS` env var; no config file; self-contained parallel runs
**Notable finding — qwen3-coder-30b passes bash-01:** All other models failed bash-01 when nanocoder needed execute_bash approval (blocked in headless mode). qwen3-coder-30b is the only model that wrote a correct shell script without needing chmod, scoring 4/4. This reveals a model-level capability difference, not just an adapter limitation.

- **Status:** production-ready — 7/7 models benchmarked. Top picks: qwen3.6-35b-a3b (97%), qwen3-coder-30b (95%), gemma-4-26b-a4b-qat (95%).
